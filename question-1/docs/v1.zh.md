# v1 说明

返回主入口：[README.md](../README.md)

## 1. 为什么先做 v1

v1 解决的是最基础也最刚需的问题：给定一批 SOP HTML 文档，先把“能搜到正确文档”的能力做出来。

在这个阶段，项目只关心三件事：

1. 把 HTML 中真正可见的文本抽出来
2. 建立稳定的关键词检索基线
3. 提供最小可用的 API 和页面，方便人工验证

如果连 v1 都不稳定，后面的语义搜索和 Agent 都没有可靠的 grounding 基础。

## 2. v1 的能力边界

v1 当前具备：

- 通过 `POST /v1/documents` 导入或替换单篇 HTML 文档
- 通过 `GET /v1/search?q=...` 做关键词搜索
- 通过 `GET /v1` 打开一个最小搜索页面
- 自动忽略 `script`、`style`、`noscript` 等非可见内容
- 处理中英文、全角字符和部分技术符号，例如 `API/v1`、`&`

v1 当前不具备：

- 语义理解
- 会话能力
- 外部执行器
- 持久化存储

## 3. 需求与设计取舍

v1 的核心设计点：

- 只索引可见文本，不让 `script` 或 `<head>` 噪音污染结果
- 用内存态文档存储和 BM25 风格词法检索，保证实现简单、可读、可测
- 保持 API 响应结构稳定：`query` + `results[]`

为什么这样设计：

- 面试项目优先要可解释，而不是先上重型搜索基础设施
- v1 是后续 v2 和 v3 的“稳态基线”
- 内存方案足以覆盖当前 demo 语料规模

## 4. API 列表

| 方法 | 路径 | 说明 | 是否鉴权 |
| --- | --- | --- | --- |
| `GET` | `/v1` | 关键词搜索页面 | 否 |
| `GET` | `/v1/search?q=...` | 关键词搜索 API | 否 |
| `POST` | `/v1/documents` | 导入或替换文档 | 是，若配置了 `API_KEY` |

## 5. 请求与响应示例

### 5.1 导入文档

请求：

```bash
curl -X POST 'http://127.0.0.1:8000/v1/documents' \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: dev-secret' \
  -d '{
    "id": "sop-custom",
    "html": "<html><head><title>Custom SOP</title></head><body><p>Primary &amp; backup path is active.</p></body></html>"
  }'
```

请求体：

```json
{
  "id": "sop-custom",
  "html": "<html><head><title>Custom SOP</title></head><body><p>Primary &amp; backup path is active.</p></body></html>"
}
```

响应体：

```json
{
  "id": "sop-custom",
  "title": "Custom SOP"
}
```

说明：

- 同一个 `id` 再次写入时，会按“替换”语义更新文档
- 当前实现首次写入与替换写入都返回 `201`

### 5.2 搜索文档

请求：

```bash
curl 'http://127.0.0.1:8000/v1/search?q=OOM'
```

响应体示例：

```json
{
  "query": "OOM",
  "results": [
    {
      "id": "sop-001",
      "title": "后端服务 On-Call SOP",
      "snippet": "服务 OOM 时，先检查实例内存使用、最近发布记录和降级开关。",
      "score": 1.2345
    }
  ]
}
```

空查询示例：

```bash
curl 'http://127.0.0.1:8000/v1/search?q='
```

响应体：

```json
{
  "query": "",
  "results": []
}
```

### 5.3 页面访问

```bash
curl 'http://127.0.0.1:8000/v1'
```

说明：

- 返回 HTML 页面，不是 JSON
- 页面会调用 `GET /v1/search`

## 6. 典型使用场景

- 精确关键词已知，例如 `OOM`、`CDN`、`DNS`
- 想快速验证 HTML 解析和可见文本抽取是否正确
- 需要导入一篇临时 SOP 进行本地实验
- 需要一个稳定的检索基线，和 v2 的语义检索做对照

## 7. 错误码

| 状态码 | 场景 | 说明 |
| --- | --- | --- |
| `200` | `GET /v1/search`、`GET /v1` | 搜索成功或页面渲染成功 |
| `201` | `POST /v1/documents` | 文档写入成功 |
| `401` | `POST /v1/documents` | 配置了 `API_KEY` 但未提供正确的 `X-API-Key` |
| `422` | `POST /v1/documents` | 请求体缺字段或字段为空 |
| `500` | 任意接口 | 未处理异常，由 observability middleware 返回统一 JSON |

`500` 响应示例：

```json
{
  "detail": "Internal Server Error",
  "request_id": "generated-request-id",
  "trace_id": "generated-trace-id"
}
```

## 8. 已知限制

- 只能做词法匹配，无法理解“服务器挂了”这类非精确表述
- 数据默认保存在内存中，重启会回到 `data/` 目录初始状态
- 没有删除接口，也没有持久化数据库
- `POST /v1/documents` 会立即更新 v1 和 v2 的内存索引，但不会自动刷新 v3 的 `catalog.json`
- 当前没有分页与多租户隔离

## 9. 后续可扩展点

- 引入持久化存储与文档版本管理
- 增加删除、批量导入、重建索引接口
- 增加分页、过滤和更丰富的 snippet 高亮
- 给写接口增加审计记录
- 在 v1 上引入更细粒度的文档元数据过滤

## 10. 为什么要演进到 v2

v1 的最大限制是“用户怎么问，文档里就得怎么写”。当用户说：

- `服务器挂了`
- `机器学习模型出问题`
- `黑客攻击`

这些表述未必会按原词出现在 SOP 里。于是就需要 v2：保留 v1 基线，再增加 chunk 级语义检索与 hybrid fusion。
