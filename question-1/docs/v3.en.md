# v3 Guide

Back to the main entry: [README.en.md](../README.en.md)

## 1. Why the project evolves from v2 to v3

v2 can already find relevant SOPs fairly well, but the user still has to do the final steps manually:

1. open the document
2. locate the relevant section
3. turn that into an action recommendation
4. remember which files were consulted

v3 turns that chain into a constrained but transparent chat-style assistant.

The important words are:

- constrained: not arbitrary tools, files, or free-form planning
- transparent: return tool traces and `consulted_files`
- grounded: answers should come from files that were actually read

## 2. What v3 can and cannot do

Current v3 capabilities:

- `GET /v3`: chat page
- `POST /v3/chat`: chat API
- `GET /v3/history/{session_id}`: fetch the current in-memory chat history
- catalog-first routing through `catalog.json`
- exactly one tool: `readFile(fname)`
- structured tool traces in `tool_calls`
- truthful file-level provenance in `consulted_files`
- full turn history in `history`
- minimal multi-turn behavior via `session_id`
- an `X-API-Key` input on the page plus a rendered message timeline
- default rule-based `AgentLoop` when `OPENAI_API_KEY` is not set

What v3 does not do:

- orchestrate multiple tools
- browse directories
- persist sessions
- execute external actions
- apply global shared rate limits across multiple instances

## 3. Core design trade-offs

### Why catalog-first

The current implementation does not let the agent guess filenames first. It always reads `catalog.json` before deciding which SOP files to open. This keeps routing:

- explainable
- easy to test
- clearly separated from the later “read file content” step

### Why a single tool

The task intentionally keeps the tool boundary narrow. The implementation only exposes:

```text
readFile(fname)
```

This keeps v3 behavior controllable:

- no directory listing
- no wildcards
- no access outside the `data/` directory
- absolute paths and path traversal are rejected

### Why return tool traces and consulted files

If the API returned only the final answer, the user could not tell:

- which files were actually read
- whether the tool boundary was respected
- whether the current turn used fresh file reads or only previous memory

That is why the API returns both:

- `tool_calls`
- `consulted_files`

## 4. API list

| Method | Path | Description | Auth |
| --- | --- | --- | --- |
| `GET` | `/v3` | chat page | No |
| `POST` | `/v3/chat` | chat API | Yes, when `API_KEY` is configured |
| `GET` | `/v3/history/{session_id}` | fetch session history | Yes, when `API_KEY` is configured |

Additional constraints:

- `/v3/chat` is rate-limited by IP
- the default limit is `30` requests per minute and is configurable through `RATE_LIMIT_PER_MIN`

### 5.3 Read session history

```bash
curl 'http://127.0.0.1:8000/v3/history/0f2a4688-8272-42f4-84b9-4dbd93b42c85' \
  -H 'X-API-Key: dev-secret'
```

Notes:

- this returns the accumulated turns for the given `session_id`
- when `API_KEY` is configured, the history endpoint also requires `X-API-Key`

### 5.4 Open the page

```bash
curl 'http://127.0.0.1:8000/v3'
```

Notes:

- this returns HTML, not JSON
- the frontend calls `POST /v3/chat`
- if the browser already has a `session_id`, the page also calls `GET /v3/history/{session_id}` to restore prior turns
- the page includes an `X-API-Key` input; when backend auth is enabled, the same key must be entered there
- this input expects the app `API_KEY`, not `OPENAI_API_KEY`

## 6. Typical use cases

- “数据库主从延迟超过30秒怎么处理？”
- “服务 OOM 了怎么办？”
- “P0 故障的响应流程是什么？”
- “怀疑有人入侵了系统”
- “推荐结果质量下降了”

These use cases share the same pattern:

- the user wants an actionable recommendation, not a result list
- the user needs to know which SOPs support the answer
- the system needs a small amount of conversational continuity

## 7. Error codes

| Status | Scenario | Meaning |
| --- | --- | --- |
| `200` | `GET /v3`, `POST /v3/chat`, `GET /v3/history/{session_id}` | successful page render, chat response, or history fetch |
| `401` | `POST /v3/chat`, `GET /v3/history/{session_id}` | `API_KEY` is configured but the request does not provide the correct `X-API-Key` |
| `404` | `GET /v3/history/{session_id}` | the session does not exist or was already lost |
| `422` | `POST /v3/chat` | `message` is missing or empty |
| `429` | `POST /v3/chat` | IP-based rate limit exceeded |
| `500` | any endpoint | unhandled exception wrapped by the observability middleware |

Rate-limit error example:

```json
{
  "detail": "rate limit exceeded"
}
```

Auth error example:

```json
{
  "detail": "unauthorized"
}
```

## 8. Known limitations

- sessions are stored in memory and disappear after a restart
- rate limiting is per-process, not globally shared across instances
- the system supports only one tool, `readFile(fname)`
- the v3 catalog is generated on startup only; documents uploaded later through `POST /v1/documents` require a restart before v3 can see them
- the default rule-based `AgentLoop` favors transparency and testability over general agent intelligence
- even when `OPENAI_API_KEY` is configured, the single-tool safety boundary still applies

## 9. Extension points

- replace the session store with Redis or a database
- refresh the catalog incrementally instead of only on startup
- add shared rate-limit storage for consistent multi-instance limiting
- add richer audit metadata such as actor, session, and routing decision details
- improve grounding and ranking while preserving the single-tool boundary

## 10. Why v3 is not a free-form autonomous agent

The goal of this project is not to build an unconstrained general agent. It is to build something that is:

- explainable
- testable
- bounded by a clear safety model
- able to show the actual tool calls behind the answer

That is why the current implementation is better described as a catalog-first constrained agent workflow rather than a fully autonomous agent.
