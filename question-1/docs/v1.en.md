# v1 Guide

Back to the main entry: [README.en.md](../README.en.md)

## 1. Why v1 exists first

v1 solves the most basic and most necessary problem first: given a set of SOP HTML files, make sure the system can reliably find the right document.

At this stage, the project focuses on three things:

1. extract the truly visible text from HTML
2. build a stable keyword-search baseline
3. expose a minimal API and page for manual verification

If v1 is not reliable, later semantic search and agent behavior do not have a trustworthy grounding layer.

## 2. What v1 can and cannot do

Current v1 capabilities:

- ingest or replace one HTML document via `POST /v1/documents`
- run keyword search via `GET /v1/search?q=...`
- serve a minimal search page via `GET /v1`
- ignore non-visible content such as `script`, `style`, and `noscript`
- handle mixed Chinese, English, full-width characters, and selected technical symbols such as `API/v1` and `&`

What v1 does not do:

- semantic understanding
- session handling
- external execution
- persistent storage

## 3. Requirements and design trade-offs

Core v1 design choices:

- index only visible text so that `script` and `<head>` noise does not pollute search
- use in-memory storage plus BM25-style lexical retrieval for a small, readable, testable system
- keep the response contract stable: `query` + `results[]`

Why this choice makes sense:

- an interview project should be explainable before it becomes infrastructure-heavy
- v1 is the stable baseline for both v2 and v3
- an in-memory approach is enough for the current demo corpus

## 4. API list

| Method | Path | Description | Auth |
| --- | --- | --- | --- |
| `GET` | `/v1` | keyword search page | No |
| `GET` | `/v1/search?q=...` | keyword search API | No |
| `POST` | `/v1/documents` | ingest or replace a document | Yes, when `API_KEY` is configured |

## 5. Request and response examples

### 5.1 Ingest a document

Request:

```bash
curl -X POST 'http://127.0.0.1:8000/v1/documents' \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: dev-secret' \
  -d '{
    "id": "sop-custom",
    "html": "<html><head><title>Custom SOP</title></head><body><p>Primary &amp; backup path is active.</p></body></html>"
  }'
```

Request body:

```json
{
  "id": "sop-custom",
  "html": "<html><head><title>Custom SOP</title></head><body><p>Primary &amp; backup path is active.</p></body></html>"
}
```

Response body:

```json
{
  "id": "sop-custom",
  "title": "Custom SOP"
}
```

Notes:

- writing the same `id` again replaces the document
- both first-time ingests and replacements return `201` today

### 5.2 Search documents

Request:

```bash
curl 'http://127.0.0.1:8000/v1/search?q=OOM'
```

Sample response:

```json
{
  "query": "OOM",
  "results": [
    {
      "id": "sop-001",
      "title": "后端服务 On-Call SOP",
      "snippet": "服务 OOM 时，先检查实例内存使用、最近发布记录和降级开关。",
      "score": 1.2345
    }
  ]
}
```

Blank-query example:

```bash
curl 'http://127.0.0.1:8000/v1/search?q='
```

Response:

```json
{
  "query": "",
  "results": []
}
```

### 5.3 Open the page

```bash
curl 'http://127.0.0.1:8000/v1'
```

Notes:

- this returns HTML, not JSON
- the page calls `GET /v1/search` under the hood

## 6. Typical use cases

- the keyword is already known, for example `OOM`, `CDN`, or `DNS`
- you want to verify HTML parsing and visible-text extraction quickly
- you want to upload a temporary SOP for a local experiment
- you need a stable retrieval baseline to compare against v2

## 7. Error codes

| Status | Scenario | Meaning |
| --- | --- | --- |
| `200` | `GET /v1/search`, `GET /v1` | successful search or page render |
| `201` | `POST /v1/documents` | successful document write |
| `401` | `POST /v1/documents` | `API_KEY` is configured but the request does not provide the correct `X-API-Key` |
| `422` | `POST /v1/documents` | missing or empty required fields |
| `500` | any endpoint | unhandled exception wrapped by the observability middleware |

Example `500` response:

```json
{
  "detail": "Internal Server Error",
  "request_id": "generated-request-id",
  "trace_id": "generated-trace-id"
}
```

## 8. Known limitations

- lexical matching only; it cannot understand paraphrases such as “服务器挂了”
- data is in memory by default, so a restart returns to the initial `data/` corpus
- there is no delete API and no persistent database
- `POST /v1/documents` updates the v1 and v2 in-memory indexes immediately, but it does not refresh the v3 `catalog.json`
- there is no pagination or multi-tenant isolation

## 9. Extension points

- add persistent storage and document versioning
- add delete, bulk-ingest, and rebuild-index APIs
- add pagination, filtering, and richer snippet highlighting
- add audit logging for write operations
- add document metadata filters on top of the lexical baseline

## 10. Why the project evolves to v2

The biggest v1 limitation is that the user must ask with words that overlap the document text. Queries such as:

- `服务器挂了`
- `机器学习模型出问题`
- `黑客攻击`

may not appear literally in the SOPs. That leads to v2: keep the v1 baseline, then add chunk-level semantic retrieval and hybrid fusion.
