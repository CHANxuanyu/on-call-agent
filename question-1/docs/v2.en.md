# v2 Guide

Back to the main entry: [README.en.md](../README.en.md)

## 1. Why the project evolves from v1 to v2

v1 already provides stable keyword search, but it has a natural weakness: the user wording does not have to match the SOP wording exactly.

For example:

- the user asks “服务器挂了”
- the SOP says “服务不可用”, “集群故障”, or “后端服务异常”

If the project keeps relying only on lexical overlap, recall becomes unstable. v2 adds semantic retrieval without throwing away the v1 baseline.

## 2. What v2 can and cannot do

Current v2 capabilities:

- semantic search via `GET /v2/search?q=...`
- a minimal semantic-search page via `GET /v2`
- section-aware HTML chunking instead of one embedding per document
- dense retrieval plus lexical fusion
- deterministic query rewrite for a small set of broad outage-style queries
- startup warmup to reduce the cost of the first real query

What v2 does not do:

- use a vector database
- use a reranker
- support multi-turn chat
- connect to external knowledge sources

## 3. Core design trade-offs

### Why chunk-level retrieval

One SOP often contains several scenarios. If the whole document is embedded as a single vector, unrelated sections get mixed together. v2 chooses section-aware chunks so that:

- results stay closer to the actual incident scenario
- snippets come from more relevant sections
- v3 can later build grounded answers from better retrieval units

### Why keep lexical fusion

v2 does not replace v1. It keeps the lexical baseline and fuses it with dense retrieval because:

- very short operational queries are not always stable under pure vector search
- exact keyword hits are still useful support signals

### Why add query rewrite

For broad outage questions such as “服务器挂了”, semantic vectors alone do not always land on the right responsibility domain. The current version only expands a narrow class of broad outage queries. Example:

- original query: `服务器挂了`
- expanded variants may include: `后端服务挂了`, `SRE 集群故障`, `服务不可用`

This is easier to test and easier to explain than LLM-based rewriting in an interview project.

## 5. Request and response examples

### 5.1 Semantic search

Request:

```bash
curl 'http://127.0.0.1:8000/v2/search?q=服务器挂了'
```

Sample response:

```json
{
  "query": "服务器挂了",
  "results": [
    {
      "id": "sop-001",
      "title": "后端服务 On-Call SOP",
      "snippet": "服务 OOM 时，先检查实例内存使用、最近发布记录和降级开关。",
      "score": 0.7421
    },
    {
      "id": "sop-004",
      "title": "SRE基础设施 On-Call SOP",
      "snippet": "集群层面故障时，先检查节点、Ingress 和核心监控告警。",
      "score": 0.7015
    }
  ]
}
```

Two more representative queries:

```bash
curl 'http://127.0.0.1:8000/v2/search?q=黑客攻击'
curl 'http://127.0.0.1:8000/v2/search?q=机器学习模型出问题'
```

### 5.2 Open the page

```bash
curl 'http://127.0.0.1:8000/v2'
```

Notes:

- this returns HTML, not JSON
- the frontend calls `GET /v2/search`

## 6. Typical use cases

- the user wording is broad or not aligned with exact SOP terminology
- you want “semantic similarity” instead of literal token overlap
- you want better recall and ranking stability on top of the v1 baseline
- you want to compare v1 and v2 manually on the same query

## 7. Optional runtime configuration

v2 exposes a set of optional environment variables for score experiments and ranking tuning:

| Environment variable | Default | Description |
| --- | --- | --- |
| `V2_SCORE_EXPERIMENT_PRESET` | `baseline` | one of `baseline`, `separation_soft`, `separation_strong` |
| `V2_QUERY_VARIANT_MERGE_STRATEGY` | `weighted_sum` | one of `weighted_sum`, `max_score`, `top2_avg` |
| `V2_DISPLAY_SCORE_TEMPERATURE` | `1.0` | display-score temperature, must be positive |
| `V2_FUSION_DENSE_WEIGHT` | `0.7` | dense-search weight |
| `V2_FUSION_LEXICAL_WEIGHT` | `0.3` | lexical-search weight |

Example:

```bash
export V2_SCORE_EXPERIMENT_PRESET=separation_soft
export V2_FUSION_DENSE_WEIGHT=0.75
export V2_FUSION_LEXICAL_WEIGHT=0.25
uvicorn app.main:app --reload
```

These settings affect v2 only. They do not change the v1 or v3 API contracts.

## 8. Error codes

| Status | Scenario | Meaning |
| --- | --- | --- |
| `200` | `GET /v2/search`, `GET /v2` | successful search or page render |
| `500` | unhandled startup or runtime exception | for example model load failure or another unexpected runtime error |

Additional notes:

- a blank query does not return `400`; it returns `200` with an empty result set
- if the semantic model cannot be loaded on the first real startup, the app may fail during startup rather than at query time

## 9. Known limitations

- the first real startup can be slow because the model may need to download
- the index is in memory and not suitable for a large corpus
- the score is a display-oriented relevance score, not a calibrated probability
- there is no reranker or offline evaluation-driven tuning loop yet
- v2 only returns search results; it does not turn them into a grounded recommendation

## 10. Extension points

- replace the in-memory semantic index with a vector database
- add a reranker or cross-encoder
- add an offline evaluation set with systematic relevance judgments
- add async prebuild or caching for startup warmup
- add richer query rewrite or intent classification

## 11. Why the project evolves to v3

Even if v2 can find the right SOP, the user still has to:

1. open the file
2. locate the relevant section
3. turn it into an actionable recommendation

v3 is not just a better result list. It connects “find files + read files + answer with grounding” into a constrained but transparent chat workflow.
