# v3 说明

返回主入口：[README.md](../README.md)

## 1. 为什么从 v2 演进到 v3

v2 已经能较稳定地找出相关 SOP，但它仍然要求用户自己完成最后几步：

1. 打开文档
2. 定位相关段落
3. 组织成处理建议
4. 记住刚刚参考过哪些文件

v3 的目标就是把这条链路收束成一个“受限但透明”的对话式助手。

这里的关键词是：

- 受限：不是任意工具、任意文件、任意推理链
- 透明：返回 tool trace 和 `consulted_files`
- grounded：回答尽量来自真实读过的文件

## 2. v3 的能力边界

v3 当前具备：

- `GET /v3`：对话页面
- `POST /v3/chat`：对话 API
- `GET /v3/history/{session_id}`：读取当前内存 session 的历史消息
- 通过 `catalog.json` 做 catalog-first 路由
- 只允许一个工具：`readFile(fname)`
- 返回工具调用轨迹 `tool_calls`
- 返回当前回答真实读取过的 SOP 文件 `consulted_files`
- 返回完整消息历史 `history`
- 用 `session_id` 做最小多轮会话
- 页面内可输入 `X-API-Key`，并渲染历史消息列表
- 在未配置 `OPENAI_API_KEY` 时，默认使用规则式 `AgentLoop`

v3 当前不具备：

- 多工具编排
- 任意目录浏览
- 持久化 session
- 外部执行动作
- 生产级多实例共享限流

## 3. 核心设计取舍

### 为什么是 catalog-first

当前实现不会让 Agent 先“盲猜文件名”，而是固定先读 `catalog.json`，再决定读哪些 SOP 文件。这样做的原因是：

- 路由依据更可解释
- 可以被测试稳定锁定
- 能把“文件选择”与“文件内容阅读”分开

### 为什么只保留一个工具

题目要求里，Agent 的工具边界非常克制。当前实现只保留：

```text
readFile(fname)
```

这使得 v3 的行为更可控：

- 不允许列目录
- 不允许通配符
- 不允许读 `data/` 目录外的路径
- 对绝对路径和 path traversal 明确拒绝

### 为什么要返回 tool trace 和 consulted_files

如果只返回最终答案，用户无法知道：

- 它到底看了哪些文件
- 是否真的按约束读了文件
- 是否当前这一轮参考的是“新读取的文件”还是“上轮记忆”

所以当前 API 会明确返回：

- `tool_calls`
- `consulted_files`

## 4. API 列表

| 方法 | 路径 | 说明 | 是否鉴权 |
| --- | --- | --- | --- |
| `GET` | `/v3` | 对话页面 | 否 |
| `POST` | `/v3/chat` | 对话接口 | 是，若配置了 `API_KEY` |
| `GET` | `/v3/history/{session_id}` | 读取会话历史 | 是，若配置了 `API_KEY` |

额外约束：

- `/v3/chat` 有按 IP 的限流
- 默认每分钟 `30` 次，可通过 `RATE_LIMIT_PER_MIN` 调整

  -d '{
    "message": "数据库主从延迟超过30秒怎么处理？"
  }'

### 5.3 读取会话历史

```bash
curl 'http://127.0.0.1:8000/v3/history/0f2a4688-8272-42f4-84b9-4dbd93b42c85' \
  -H 'X-API-Key: dev-secret'
```

说明：

- 返回指定 `session_id` 的累计历史消息
- 如果配置了 `API_KEY`，这里同样需要带 `X-API-Key`

### 5.4 页面访问

```bash
curl 'http://127.0.0.1:8000/v3'
```

说明：

- 返回 HTML 页面，不是 JSON
- 页面前端会调用 `POST /v3/chat`
- 如果浏览器里已有 `session_id`，页面会再请求 `GET /v3/history/{session_id}` 恢复历史消息
- 页面提供 `X-API-Key` 输入框；如果后端开启了 `API_KEY`，需要在页面中填入同样的值
- 这里填写的是服务自己的 `API_KEY`，不是 `OPENAI_API_KEY`

## 6. 典型使用场景

- “数据库主从延迟超过30秒怎么处理？”
- “服务 OOM 了怎么办？”
- “P0 故障的响应流程是什么？”
- “怀疑有人入侵了系统”
- “推荐结果质量下降了”

这些场景的共同点是：

- 用户想要的是“建议”而不是“结果列表”
- 用户需要知道建议来自哪些 SOP
- 系统需要保留最小会话上下文

## 7. 错误码

| 状态码 | 场景 | 说明 |
| --- | --- | --- |
| `200` | `GET /v3`、`POST /v3/chat`、`GET /v3/history/{session_id}` | 页面渲染成功、对话成功或历史读取成功 |
| `401` | `POST /v3/chat`、`GET /v3/history/{session_id}` | 配置了 `API_KEY` 但未提供正确的 `X-API-Key` |
| `404` | `GET /v3/history/{session_id}` | session 不存在或已丢失 |
| `422` | `POST /v3/chat` | `message` 缺失或为空 |
| `429` | `POST /v3/chat` | 触发按 IP 限流 |
| `500` | 任意接口 | 未处理异常，由 observability middleware 返回统一 JSON |

限流错误示例：

```json
{
  "detail": "rate limit exceeded"
}
```

鉴权错误示例：

```json
{
  "detail": "unauthorized"
}
```

## 8. 已知限制

- session 默认在内存中，进程重启后丢失
- 限流是 per-process，不是多实例共享限流
- 当前只支持一个工具 `readFile(fname)`
- v3 的 catalog 只在 startup 时生成；运行中通过 `POST /v1/documents` 导入的新文档，需要重启才能被 v3 看见
- 默认规则式 `AgentLoop` 适合透明和可测，不追求通用智能体能力
- 即使配置了 `OPENAI_API_KEY`，也仍然受单工具边界约束

## 9. 后续可扩展点

- 把 session store 换成 Redis / 数据库
- catalog 改成增量刷新，而不是只在 startup 生成
- 引入共享限流存储，实现多实例一致限流
- 增加更细粒度的审计信息，例如 actor、session、decision metadata
- 在保持单工具边界的前提下，增加更强的 grounding / ranking 逻辑

## 10. 为什么 v3 没有直接做成“自由 Agent”

因为本项目的目标不是做一个不可控的通用智能体，而是做一个：

- 可解释
- 可测
- 有安全边界
- 能展示真实工具调用过程

这也是为什么当前实现更像一个“catalog-first 的受限 agent workflow”，而不是可以任意探索环境的 autonomous agent。
