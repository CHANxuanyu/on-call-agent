# v2 说明

返回主入口：[README.md](../README.md)

## 1. 为什么从 v1 演进到 v2

v1 已经能做稳定的关键词搜索，但它有一个天然短板：用户的描述不一定和 SOP 的原文完全一致。

例如：

- 用户问“服务器挂了”
- 文档里写的是“服务不可用”“集群故障”“后端服务异常”

如果继续只靠关键词匹配，召回会不稳定。v2 的目标就是在不推翻 v1 的前提下，增加一层语义检索能力。

## 2. v2 的能力边界

v2 当前具备：

- `GET /v2/search?q=...` 语义搜索
- `GET /v2` 最小语义搜索页面
- HTML 按 section chunking，而不是整篇文档一个向量
- dense retrieval + lexical fusion
- 对少量“宽泛故障表述”做确定性 query rewrite
- 启动时自动 warmup，减少首次真实查询的冷启动成本

v2 当前不具备：

- 向量数据库
- reranker
- 会话式多轮对话
- 外部知识源接入

## 3. 核心设计取舍

### 为什么做 chunk 级检索

一篇 SOP 往往包含多个场景。把整篇文档压成一个向量会把不相关段落混在一起。v2 选择 section-aware chunk，目的是：

- 让结果更贴近具体场景
- 让 snippet 来自更相关的段落
- 为后续 v3 的 grounded answer 打基础

### 为什么保留 lexical fusion

v2 不是“把 v1 推倒重来”，而是把 v1 当稳定基线继续利用。当前实现会把 dense 结果和 v1 词法结果做小规模融合，原因是：

- 对运维类短 query，纯向量检索不一定稳定
- 关键词命中仍然是有价值的支持信号

### 为什么加入 query rewrite

对“服务器挂了”这类非常宽泛的问题，语义向量并不总能直接落到正确责任域。当前版本只对小范围宽泛故障 query 做确定性扩展，例如：

- 原始 query：`服务器挂了`
- 扩展后可能会额外搜索：`后端服务挂了`、`SRE 集群故障`、`服务不可用`

这比直接上 LLM 改写更容易测试，也更适合面试项目。

## 4. API 列表

| 方法 | 路径 | 说明 | 是否鉴权 |
| --- | --- | --- | --- |
| `GET` | `/v2` | 语义搜索页面 | 否 |
| `GET` | `/v2/search?q=...` | 语义搜索 API | 否 |

## 5. 请求与响应示例

### 5.1 语义搜索

请求：

```bash
curl 'http://127.0.0.1:8000/v2/search?q=服务器挂了'
```

响应体示例：

```json
{
  "query": "服务器挂了",
  "results": [
    {
      "id": "sop-001",
      "title": "后端服务 On-Call SOP",
      "snippet": "服务 OOM 时，先检查实例内存使用、最近发布记录和降级开关。",
      "score": 0.7421
    },
    {
      "id": "sop-004",
      "title": "SRE基础设施 On-Call SOP",
      "snippet": "集群层面故障时，先检查节点、Ingress 和核心监控告警。",
      "score": 0.7015
    }
  ]
}
```

再看两个典型 query：

```bash
curl 'http://127.0.0.1:8000/v2/search?q=黑客攻击'
curl 'http://127.0.0.1:8000/v2/search?q=机器学习模型出问题'
```

### 5.2 页面访问

```bash
curl 'http://127.0.0.1:8000/v2'
```

说明：

- 返回 HTML 页面，不是 JSON
- 页面前端会调用 `GET /v2/search`

## 6. 典型使用场景

- 用户表述较宽泛，不知道准确 SOP 术语
- 希望按“语义相近”而不是“原词命中”找文档
- 希望在 v1 之上提高召回与排序稳定性
- 想手动比较 v1 和 v2 对同一 query 的差异

## 7. 可选运行时配置

v2 暴露了一组可选环境变量，主要用于评分实验与排序调参：

| 环境变量 | 默认值 | 说明 |
| --- | --- | --- |
| `V2_SCORE_EXPERIMENT_PRESET` | `baseline` | 可选 `baseline`、`separation_soft`、`separation_strong` |
| `V2_QUERY_VARIANT_MERGE_STRATEGY` | `weighted_sum` | 可选 `weighted_sum`、`max_score`、`top2_avg` |
| `V2_DISPLAY_SCORE_TEMPERATURE` | `1.0` | display score 温度，必须为正数 |
| `V2_FUSION_DENSE_WEIGHT` | `0.7` | dense 结果权重 |
| `V2_FUSION_LEXICAL_WEIGHT` | `0.3` | lexical 结果权重 |

示例：

```bash
export V2_SCORE_EXPERIMENT_PRESET=separation_soft
export V2_FUSION_DENSE_WEIGHT=0.75
export V2_FUSION_LEXICAL_WEIGHT=0.25
uvicorn app.main:app --reload
```

这些配置只影响 v2，不会改变 v1 或 v3 的接口结构。

## 8. 错误码

| 状态码 | 场景 | 说明 |
| --- | --- | --- |
| `200` | `GET /v2/search`、`GET /v2` | 搜索成功或页面渲染成功 |
| `500` | 启动或查询过程中的未处理异常 | 例如模型加载失败、意外运行时错误 |

补充说明：

- 空查询不会返回 `400`，而是 `200` + 空结果
- 如果语义模型首次加载失败，应用可能在 startup 阶段直接失败，而不是等到某次查询时才报错

## 9. 已知限制

- 首次真实启动可能因为模型下载而变慢
- 索引是内存态，不适合大规模文档集
- score 是“便于展示的相关性分数”，不是严格校准后的概率
- 当前没有 reranker，也没有离线评估集驱动的自动调参
- v2 只返回搜索结果，不负责整合成面向用户的建议

## 10. 后续可扩展点

- 把 in-memory semantic index 换成向量数据库
- 增加 reranker 或 cross-encoder
- 增加离线评测集与更系统的 relevance 标注
- 为 startup warmup 增加异步预构建或缓存
- 增加更细粒度的 query rewrite / intent classification

## 11. 为什么要演进到 v3

即使 v2 已经能把对的 SOP 找出来，用户仍然需要自己：

1. 打开文档
2. 定位相关段落
3. 组织成处理建议

v3 的目标不是再做一个更强的搜索结果列表，而是把“找文件 + 读文件 + 返回 grounded answer”串成一个受限但透明的对话流程。
